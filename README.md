## Внутренняя очередь для фоновой обработки входящих задач

### Что нужно сделать?

1. Принимать задания через `POST /enqueue`
    • Тело: JSON `{"id":"<string>","payload":"<string>","max_retries":<int>}`.
    • Задание попадает в буферизированную очередь (размер — из конфигурации).
    • Авторизация не требуется.
2. Обрабатывать задания пулом воркеров (`WORKERS` настраивается)
    • Каждое задание «работает» 100–500 мс (симулируем обработку).
    • 20% задач «падают» (симулируем ошибки) → должен быть экспоненциальный бэкофф с джиттером и до `max_retries` повторов.
    • Хранить и обновлять состояние: `queued` | `running` | `done` | `failed`.
3. Отдавать healthcheck
    • `GET /healthz` → 200 OK.
4. Корректно завершать программу по SIGINT/SIGTERM
    • Перестаем принимать новые задачи.
    • Дожидаемся завершения текущих.

### Ограничения
• Версия Go 1.25.
• Конфигурация через переменные окружения: `WORKERS` (по умолчанию 4), `QUEUE_SIZE` (по умолчанию 64).

---

### Ход решения задачи:

1) **Конфигурация:** Реализована загрузка параметров `WORKERS` и `QUEUE_SIZE` через переменные окружения (поддерживается чтение из `.env` файла), а также предусмотрена возможность конфигурации через `config.yaml`.
2) **Роутинг:** Используется стандартный маршрутизатор Go 1.22+ (`http.ServeMux`) с явным указанием методов для обработки `POST /enqueue` и `GET /healthz`.
3) **Запуск сервера:** HTTP-сервер слушает порт `8080` в отдельной горутине, не блокируя основной поток, который ожидает системных сигналов.
4) **Graceful Shutdown:** Перехватываем сигналы `SIGINT` и `SIGTERM`. При их получении:
    • Сервер перестает принимать новые HTTP-запросы (дается таймаут 10 секунд на завершение текущих соединений).
    • Пул воркеров получает сигнал на остановку через `context.Context` и безопасно завершает все активные задачи в очереди.
5) **Healthcheck:** `GET /healthz` возвращает HTTP-статус `200 OK` и JSON `{"status":"OK"}`.
6) **Обработка задач (POST /enqueue):** Принимает JSON, валидирует параметры и передает задачу в пул воркеров. Если очередь не переполнена, клиенту возвращается статус `202 Accepted` и тело задачи с обновленным статусом `queued`. В случае переполнения пула возвращается `503 Service Unavailable`.
7) **Бизнес-логика (Runner):** В методе обработки вычисляется экспоненциальный бэкофф с джиттером до `max_retries` повторов. Задание симулирует полезную работу (100–500 мс). Для демонстрации механизма повторов искусственно внедрен шанс ошибки (20%). Статусы задачи обновляются в реальном времени.
8) **Worker Pool:** Реализован надежный пул воркеров без состояния гонки (Race Conditions). 
    • `Submit(ctx, task)` — добавляет задачу в буферизированный канал с поддержкой отмены по контексту. 
    • `Stop(ctx)` — закрывает канал с задачами. Воркеры дорабатывают оставшиеся в буфере задания и завершают свои горутины, а `sync.WaitGroup` гарантирует, что программа не завершится до их полной остановки.
9) **Логирование:** Использован современный структурированный логгер `log/slog` для удобного отслеживания жизненного цикла каждой задачи (по `task_id`) и системных событий.
